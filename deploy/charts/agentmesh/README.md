# AgentMesh Helm Chart

Deploys the [AgentMesh](https://github.com/vigneshvn1995/agentmesh) LLM proxy gateway onto a Kubernetes cluster. Optionally bundles **Redis** (Bitnami) for distributed token budget enforcement and **Qdrant** for semantic caching via subchart dependencies, so the entire proxy + state backend stack can be spun up with a single `helm install`.

---

## Prerequisites

| Tool | Version |
|---|---|
| Helm | 3.10+ |
| Kubernetes | 1.25+ |
| kubectl | matching cluster version |

---

## Quick Start (Sandbox)

The fastest path to a fully functional evaluation environment — proxy, semantic cache, and budget tracker — with no external services required.

```bash
# 1. Pull the subchart dependencies declared in Chart.yaml.
helm dependency update deploy/charts/agentmesh

# 2. Install the full stack: AgentMesh + bundled Redis + bundled Qdrant.
helm install agentmesh deploy/charts/agentmesh \
  --set redis.enabled=true \
  --set qdrant.enabled=true \
  --set agentmeshConfig.cache.enabled=true \
  --set agentmeshConfig.redis.address="agentmesh-redis-master:6379"
```

`agentmesh-redis-master` is the Kubernetes service name auto-generated by the Bitnami Redis subchart when the Helm release is named `agentmesh`. Adjust the prefix if you use a different release name.

### Verify the deployment

```bash
kubectl rollout status deployment/agentmesh

# Proxy port
kubectl port-forward svc/agentmesh 8080:8080 &

# Send a test request (replace the key with the one in your agentmeshConfig)
curl http://localhost:8080/v1/chat/completions \
  -H "Authorization: Bearer am_live_replace_me" \
  -H "Content-Type: application/json" \
  -d '{"model":"gpt-4o","messages":[{"role":"user","content":"ping"}]}'
```

## Installing in Production

For a production deployment you will supply your own external Redis and Qdrant instances and inject credentials via Kubernetes Secrets.

```bash
helm install agentmesh deploy/charts/agentmesh \
  --namespace agentmesh \
  --create-namespace \
  --values my-production-values.yaml
```

`my-production-values.yaml` example:

```yaml
replicaCount: 3   # overridden by HPA once autoscaling kicks in

agentmeshConfig:
  version: v1
  server:
    proxy_port: 8080
    admin_port: 9090
  tenants:
    - tenant_id: acme-corp
      api_key: "am_live_acme_abc123"
      upstream_url: "https://api.openai.com/v1"
      # Do NOT put the real key here — use extraEnv below
      upstream_api_key: "REPLACED_AT_RUNTIME"
  guardrails:
    enabled: true
    loop_detection:
      window_size: 5m
      max_identical_hash: 3
  budget:
    per_agent_daily_usd: 5.00
    per_tenant_daily_usd: 200.00
    request_timeout: 30s
    tokens_per_usd: 1000
  redis:
    address: "my-redis.example.com:6379"
    failure_mode: fail-open
  cache:
    enabled: true

redis:
  enabled: false   # external Redis; subchart not needed

qdrant:
  enabled: false   # external Qdrant; subchart not needed

extraEnv:
  - name: OPENAI_API_KEY
    valueFrom:
      secretKeyRef:
        name: agentmesh-secrets
        key: openai-api-key
  - name: QDRANT_ENDPOINT
    value: "my-qdrant.example.com"
  - name: QDRANT_API_KEY
    valueFrom:
      secretKeyRef:
        name: agentmesh-secrets
        key: qdrant-api-key
```

## Configuration Rollouts

### How the ConfigMap checksum works

Every change to `agentmeshConfig` in `values.yaml` automatically triggers a rolling restart of the Deployment — no manual `kubectl rollout restart` required.

This is implemented via a pod annotation in `templates/deployment.yaml`:

```yaml
annotations:
  checksum/config: {{ include (print $.Template.BasePath "/configmap.yaml") . | sha256sum }}
```

Helm recomputes this SHA-256 digest at every `helm upgrade`. If `agentmeshConfig` changed, the rendered ConfigMap template changes, the checksum changes, the pod annotation changes, and Kubernetes detects that the pod spec has changed — triggering a rolling restart of every pod in the Deployment.

**Practical consequence:** You never need to manually cycle pods after a config change. Run:

```bash
helm upgrade agentmesh deploy/charts/agentmesh --reuse-values \
  --set agentmeshConfig.budget.per_agent_daily_usd=10.00
```

The Deployment will automatically roll to pick up the new budget value within seconds.

## Security Posture

### Container hardening

The default `values.yaml` enforces a minimal privilege posture out of the box:

```yaml
podSecurityContext:
  runAsNonRoot: true
  runAsUser: 1000
  fsGroup: 1000

securityContext:
  allowPrivilegeEscalation: false
  readOnlyRootFilesystem: true
  capabilities:
    drop:
      - ALL
```

`readOnlyRootFilesystem: true` means the container binary cannot write to the filesystem at runtime. AgentMesh has no disk writes — all state is stored in Redis (budgets) and Qdrant (vectors) — so this incurs no operational cost while eliminating an entire class of runtime exploit.

The ConfigMap volume is mounted with `readOnly: true`:

```yaml
volumeMounts:
  - name: config
    mountPath: /etc/agentmesh
    readOnly: true
```

The config file is immutable at runtime. Any configuration change goes through Helm, which triggers a controlled rolling restart rather than a live mutation.

### Never hardcode upstream API keys

The `agentmeshConfig.tenants[].upstream_api_key` field in `values.yaml` is intentionally marked `"sk-replace-me"`. Putting real credentials directly in a values file risks them appearing in Helm release history (`helm get values agentmesh`) or being committed to version control.

The correct pattern is to store the key in a Kubernetes Secret and inject it as an environment variable that the AgentMesh binary reads at startup. Add an entry to `extraEnv` in your values override:

```yaml
extraEnv:
  - name: UPSTREAM_API_KEY_ACME
    valueFrom:
      secretKeyRef:
        name: agentmesh-tenant-secrets
        key: acme-upstream-key
```

Then reference the environment variable name in `agentmeshConfig` instead of the literal key value. A future V2 enhancement will add explicit `$ENV_VAR` interpolation support to the config loader; in V1 this pattern works when the upstream provider accepts the literal string as a no-op (use a dummy placeholder key in config, inject the real key via the standard `OPENAI_API_KEY` env var that the proxy picks up).

Create the Secret with:

```bash
kubectl create secret generic agentmesh-tenant-secrets \
  --namespace agentmesh \
  --from-literal=acme-upstream-key="sk-real-openai-key-here"
```

Rotate without restarting the pods by updating the Secret — the next request cycle picks up the new value automatically.

## Autoscaling

### HPA and `replicaCount` interaction

When `autoscaling.enabled=true` (the default), the `replicas:` field is **intentionally omitted** from the Deployment spec:

```yaml
spec:
  {{- if not .Values.autoscaling.enabled }}
  replicas: {{ .Values.replicaCount }}
  {{- end }}
```

If `replicas:` were present, every `helm upgrade` would reset the replica count back to `replicaCount: 2` — overwriting whatever the HPA had scaled to and creating a fight between Helm and the Kubernetes autoscaler. By omitting the field, Helm leaves `replicas` under the sole control of the HPA after the initial install.

`replicaCount` still serves a purpose when `autoscaling.enabled=false`: it sets the static replica count directly. It also communicates the intended minimum scale to reviewers reading the values file.

### HPA configuration

```yaml
autoscaling:
  enabled: true
  minReplicas: 2
  maxReplicas: 10
  targetCPUUtilizationPercentage: 80
```

The HPA targets `autoscaling/v2` (Kubernetes 1.23+), which supports multiple metric types. The default configuration scales on CPU utilization. To add memory-based scaling, append to your values override:

```yaml
# In my-production-values.yaml — extend the HPA metrics list
autoscaling:
  enabled: true
  minReplicas: 2
  maxReplicas: 20
  targetCPUUtilizationPercentage: 70
```

And extend `templates/hpa.yaml` with a memory metric block if needed for your workload profile.

`minReplicas: 2` enforces high availability across at least two pods at all times. Combined with the default `podAntiAffinity` rule in `values.yaml`, the scheduler will prefer to place those two pods on different nodes.
